# Code Review 與 GitFlow — 正規開發流程初體驗

> 一個非軟體開發背景的獨立開發者，導入類正規化開發流程的初體驗。
> 日期：2026-03-10

---

## 背景：為什麼突然想做這件事？

這套 CRM 系統，最初只是為了解決公司裡散落各處的紙本與 Excel 檔案。
在沒有其他技術人力的情況下，前期的開發最高指導原則只有一個：「最快速度把功能生出來，讓使用者驗證」。

那陣子，我每天的日常就是：寫 code、存檔、直接 `git push` 蓋上主分支。
完全沒有寫測試的餘裕，更別提什麼 Code Review。

**但就在最近，當系統即將交由 IT 進行內部測試時，我停下來了。**

因為當我準備使用 AI (Claude Code) 來輔助進行中型重構時，我突然意識到一個很現實的問題：
萬一我不小心把已經會動的核心邏輯「修壞了」怎麼辦？如果在沒有保護網的情況下貿然更動，那我之前的努力不就全毀了？

我意識到，不能再像前 45 天那樣「閉著眼睛直接 Push」了。

於是，我在昨天緊急利用另一台 VM 搭配 Docker 架設了隔離的實驗環境，並串接了獨立的測試資料庫。
有了這個安全的「沙盒」，我終於能放下心裡的石頭，正式導入業界標準的 **GitFlow (Feature Branch + Pull Request) 開發模式**，讓接下來的每一行修改都能安全著陸。

## 一天之內的 8 輪自我審查

有了沙盒環境後，我決定做一件大膽的事：**把自己寫的 codebase 當成別人的專案，無情地去稽核它。**
我花了一整天，刻意切換了 8 個不同的視角來審查這套系統：

### Review #1 — API 安全性稽核
審查全部 **55 個 API 端點**的認證、授權與輸入驗證。
- **發現**：稽核日誌對 readonly 使用者暴露 IP、`500` 錯誤回傳 Python 內部堆疊、各 router 的 IP 取得方式不一致
- **修復**：將 `str(e)` 替換為通用錯誤訊息，將 `_get_client_ip()` 提取為共用工具函數

### Review #2 — 合併邏輯完整性
審查 3 個合併方法（聯絡人 / 公司 / 集團）的資料一致性。
- **發現**：TOCTOU（Time-of-Check-Time-of-Use）競爭條件 — 合併執行時不會重新驗證記錄是否仍為 active
- **發現**：聯絡人合併後缺少 companies 快取清除（公司統計數過期）

### Review #3 — 快取失效完整性
稽核每一個寫入操作，驗證快取是否正確清除。
- **建立完整的快取失效矩陣**，對應每個 CRUD 方法需要清除的快取
- **發現 5 個缺口**：restore 操作缺少跨 collection 快取清除，LINE Bot 操作完全跳過快取

### Review #4 — 前端錯誤處理
審查全域錯誤處理器、Axios 攔截器、13 個頁面元件。
- **發現**：每次 API 失敗都顯示**兩次錯誤訊息**（攔截器 toast + 頁面層 toast）
- **發現**：401 處理沒有清除 Firebase Auth 狀態，造成無限重導迴圈
- **發現**：Token 刷新失敗時，靜默送出無認證請求

### Review #5 — Firestore 查詢效能
分析所有資料庫查詢的 N+1 問題和全表掃描。
- **影響估算**：聯絡人搜尋每次產生 **2,700+ 次 Firestore 讀取**（應該只要 ~3 次）
- **識別出**：4 個嚴重的 N+1 模式、5 個繞過快取的全表掃描
- **預計節省**：優化後可大幅消除無效的 Firestore 讀取次數與成本

### Review #6 — 資料驗證
稽核所有 Pydantic 模型的輸入驗證缺口。
- **發現**：所有字串欄位都沒有 `max_length`（可能被濫用塞入巨量資料）、批次操作沒有陣列數量上限、聯絡人 email 缺少格式驗證（公司有做，聯絡人沒有）

### Review #7 — 程式碼重複（DRY 分析）
系統性掃描後端和前端的複製貼上模式。
- **發現約 1,425 行**冗餘程式碼，分佈在 9 個重複模式中
- **影響最大**：3 個列表頁（聯絡人 / 公司 / 集團）共享 11 個相同模式（約 800 行重複）

### Review #8 — LINE Bot / Webhook
審查整個訊息處理管線。
- **發現**：OpenAI 同步呼叫**阻塞整個 async 事件迴圈** 5-30 秒（OCR 期間所有 API 端點凍結）
- **發現**：OpenAI API 呼叫沒有 timeout（最壞情況：應用凍結 10 分鐘）
- **修復**：遷移到 `AsyncOpenAI` 並加入可設定的 timeout

## 紀念性的第一個 Pull Request

找出這 78 個潛在優化點後，我強忍住「直接在 main 分支改一改就推上去」的原始衝動。
這一次，我逼自己走了一趟正規的流程。

### 我遵循的流程

```
1. 建立 feature branch     → refactor/extract-compute-changes
2. 做集中的修改             → 提取共用工具函數
3. 撰寫單元測試             → 15 個測試案例，涵蓋邊界情況
4. 推送到遠端               → git push origin refactor/extract-compute-changes
5. 部署到測試環境            → 在 staging 環境驗證
6. 建立 Pull Request        → PR #1，附摘要 + 測試結果
7. Squash Merge 到主分支    → 保持乾淨的 commit 歷史
8. 清理                     → 刪除 feature branch
```

### 這次 PR 具體解決了什麼？（Code Review Issue #59）

`_compute_changes()` 這個用來計算資料差異的底層函數，在初期趕工時被我直接複製貼上到了 3 個不同的 Router 裡。邏輯完全相同，只是欄位對照表不一樣。在這次 PR 中，我動刀把它抽了出來：
1. 提取到 `app/utils/changes.py`，接受可設定的 `field_mapping` 參數
2. 撰寫 15 個單元測試，涵蓋：基本變更、無變更、巢狀物件、陣列欄位、None 處理、空字串
3. 將 3 份複製替換為共用函數

**修改前**：210 行分散在 3 個檔案（70 行 × 3）
**修改後**：1 個檔案 70 行 + 3 個一行 import

## 當天順手殲滅的 15 個技術地雷

除了正式發出 PR 的重構外，我在跑完 Review 後，也立刻在沙盒環境把 **15 個小型但致命的問題**給修復了：

| 類別 | 修復數 | 主要變更 |
|------|--------|---------|
| **安全性** | 4 | 通用錯誤訊息、一致的 IP 取得、自我保護檢查 |
| **可靠性** | 4 | Async OpenAI、可設定 timeout、安全的推播通知、啟動時環境變數驗證 |
| **資料完整性** | 3 | 快取失效缺口、OCR 缺少的欄位寫入 |
| **程式碼品質** | 2 | 移除死碼、Literal 型別強制 |
| **測試** | 2 | image handler 單元測試、compute_changes 單元測試 |

## 把 AI 的建議，變成自己的防呆 SOP

身為一個初次接觸正規流程的開發者，要在腦海中記住所有 GitFlow 細節是不現實的。所以我請 AI 根據業界最佳實踐，為這個專案梳理了一份 **Feature Branch 開發流程指南**。

我把這些建議內化後記錄下來。這不只是寫給未來潛在的協作者看，更是寫給「未來的自己」的防呆清單：

- **何時要開 branch**：基於風險的決策矩陣（多檔案重構 → 必須開分支；一行修復 → 可直接推）
- **分支命名**：`feat/`、`fix/`、`refactor/`、`docs/` 前綴
- **PR 範本**：摘要、測試方式、影響範圍、Code Review 追蹤
- **合併策略**：Squash-and-merge 保持乾淨歷史
- **測試環境部署**：建 PR 前先一鍵部署到測試環境驗證
- **合併後清理**：將測試環境恢復到主分支、刪除 feature branch

## 反思：關於這次「類正規化」初體驗

**【寫出「會動」的程式很爽，但寫出「能交代」的程式很難】**

我是 Google Apps Script 和 Excel VBA 背景出身的——在那個世界裡，寫一個函數能跑、能解決當下的問題，工作就算結束了。沒有版本控制、沒有測試環境，更沒有所謂的 Code Review。

這 45 天的核心開發期，我也一直帶著這種「快速產出」的癮頭在衝刺。
但就在昨天，當我為了今天的報告準備架設沙盒環境、對著螢幕盤點這 633 個 commit 的時候，我心裡突然卡了一下。

那種感覺不是技術做不到，
而是你會意識到，之前那些「能跑就好」的自信，其實是建立在某種脆弱的假設上。

### 沒踩過坑，真的不知道痛在哪裡

當我讓 AI (Claude Code) 幫我跑完全面的 Code Review，甚至親自嘗試為 `_compute_changes()` 補寫第一行單元測試的時候，我才驚覺一件事：**原來這套系統裡，藏著這麼多我和 AI 都沒預料到的邊界情況。**

因為專案初期大量依賴 AI 協作產出，其實很多邏輯只滿足了「直觀上的 Happy Path」。當我面對一個表面能動的龐大系統，如果在沒有保護網的情況下貿然嘗試中型重構，牽一髮就會動全身。

單元測試和 Code Review 對我來說，不再是應付流程的打卡，而是強迫我從 AI 手中「重新奪回每一行程式碼絕對控制權」的過程。
但昨天在合併那第一個 Pull Request 的瞬間，我才懂了——
這不是在浪費時間，而是在幫自己裝一個**「開發者的保險絲」**。


那個「分支、測試、合併」的流程，其實是強迫一個習慣了「求快」的開發者冷靜下來，做最後一關的自我審查。這就是我學到最重要的一件事：寫測試不只是為了防呆，它其實是一種「設計工具」，它會逼你去看那些你原本想裝作沒看到的邊界情況。

### 那些被量化的「技術債」

1,425 行的重複代碼、每次查詢多送出幾千次的無效資料庫讀取。
這對我來說不再是模糊的感覺，
而是一張真實的待辦清單。

看著這張清單，我其實感覺踏實很多。
因為我知道「優化」不再是打高空，而是有具體的數字可以追蹤。

**如果時間能重來，我會對第一天的自己說：**
*   **從第一天就開始寫測試**：不還債的代價，就是要在最後一關還得通宵補課。
*   **養成開分支的肌肉記憶**：不管修多小的 bug，都要讓它在沙盒裡過一遍。
*   **把重複發生的糾錯交給機器**：設定好 pre-commit hooks，不要再浪費精氣神在排版這種事情上。

這 45 天教會了我如何「寫」程式，
而這一天的初體驗，教會了我如何把程式寫成一個真正的「產品」。
