# 作品案例集 - STAR Method

> 企業聯絡人管理系統（CRM）| 獨立全端開發 | 45 天

---

## 案例一：LINE Bot AI 名片掃描 — 高效架構與系統整合

### Situation（情境）
公司業務人員在外拜訪客戶時，收到大量紙本名片。回到辦公室後需要手動將名片資訊逐一輸入系統，一張名片平均花 3-5 分鐘，且經常出現錯字、格式不一致、重複建檔等問題。公司有 3,000+ 筆聯絡人，每週新增 10-20 張名片。

### Task（任務）
設計一套「拍照即建檔」的解決方案，讓業務人員在現場用手機拍下名片，系統能自動辨識結構化資訊並無縫存入資料庫，同時具備防呆機制避免重複建檔。

### Action（行動）

**技術選型**：
- 使用 LINE Bot 作為入口（業務人員手機已有 LINE，零學習成本）
- 使用 AI Vision API 進行 OCR（非傳統 OCR，能處理設計感強的名片）
- 自行設計 Flex Message UI（LINE 原生卡片介面）

**核心實作**：

1. **AI OCR 辨識引擎**
   - 設計結構化提示詞，要求 AI 返回 JSON 格式
   - 支援 18 個欄位智慧提取：姓名、公司、統編、職稱、電話、手機、Email、地址等
   - 內建信心分數機制，低於 50% 的欄位不採納
   - 台灣特有格式處理：市話/手機自動區分、+886 國際碼轉換、郵遞區號提取

2. **智慧重複比對**
   - 設計「公司+姓名」組合比對策略（同名但不同公司 → 不同人）
   - 電話正規化後比對（02-1234-5678 = 0212345678 = +886-2-1234-5678）
   - 找到重複時，用差異卡片展示新舊資料對比，讓使用者決定

3. **互動式確認流程**
   ```
   拍照 → AI 辨識 → 預覽卡片（可修改）→ 重複檢查 → 確認寫入
   ```
   - Quick Reply 按鈕讓使用者能快速編輯特定欄位
   - Carousel 卡片同時展示多個重複候選人

**架構決策**：
- 為什麼用 LINE 不用 Web App？→ 業務在外面沒有電腦，LINE 最順手
- 為什麼用 AI Vision 不用 Google OCR？→ 名片排版千變萬化，傳統 OCR 準確率不足
- 為什麼做重複比對而非直接建立？→ 業務常拿到同一人多次名片

### Result（成果）
- **約 3 天完成** 核心辨識引擎與 LINE Bot 的系統整合
- 實測名片建檔時間從 **3-5 分鐘降至 10 秒**
- 成功打通「現場拍照 → 系統建檔」的自動化流程，具備隨時上線的成熟度

## 案例二：資料庫遷移 — 從 ERP 到雲端，零停機

### Situation（情境）
系統最初建立在 ERP 平台（Odoo）之上。隨著前端應用擴展（LINE Bot、React Web App），我們面臨了兩大瓶頸：一是以 Odoo 為主體的雙重認證（Auth Friction）造成員工需管理兩套帳密；二是 Odoo 的原生視圖過於僵化，無法滿足營運主管對客製化報表 UI 的需求。

此外，早期的系統稽核日誌存放在容器內的 SQLite，容器重啟後日誌會被重置消失。

### Task（任務）
將核心聯絡人資料與前後端解耦，遷移至雲端資料庫（Firestore），解決認證摩擦並完全解放前端 UI 開發彈性，同時：
- 不能停機（正式環境每天都有人使用）
- 舊有資料必須無損遷移
- 確保稽核日誌獲得永久且可靠的儲存方案

### Action（行動）

**遷移策略：漸進式，非 Big Bang**

1. **Schema 設計**
   ```
   /groups          21 筆（集團）
   /companies    1,014 筆（公司）
   /contacts     1,784 筆（聯絡人）
   /projects     2,019 筆（專案）
   /project_contacts    （多對多關聯）
   /audit_logs          （稽核日誌，新建）
   ```
   - 設計 ID 映射規則：ERP ID → Firestore Document ID
   - 保留原始 ID 以便回溯

2. **遷移腳本**
   - 批次讀取 ERP 資料 → 清洗轉換 → 寫入 Firestore
   - 處理欄位名稱轉換（ERP 使用 `x_` 前綴）
   - 處理關聯資料的一致性

3. **雙軌運行期**
   - 環境變數控制資料來源：`DATA_SOURCE=firestore` / `DATA_SOURCE=odoo`
   - API 層維持完全相同的介面
   - 逐步切換，先切讀取，再切寫入

4. **前端狀態同步與快取優化**
   - 導入「樂觀更新 (Optimistic UI)」：操作瞬間重繪畫面，隱藏 API 延遲
   - 設計自癒快取 (Self-Healing Cache)：透過全域攔截器自動清理被刪除的「幽靈狀態」
   - 經由前端快取防護，Firestore 讀取次數從每天 26 萬次降至 1 萬次以下

5. **稽核日誌遷移**
   - SQLite → Firestore（永久儲存，不設自動清理）
   - 確保所有資料合併操作皆可透過日誌腳本反向溯源與還原
   - 編寫歷史資料遷移腳本

**關鍵決策**：
- 為什麼選 Firestore？→ 全託管、自動備份、即時讀寫、免運維
- 為什麼不一次全切？→ 降低風險，出問題可以立刻切回
- 為什麼保留 ERP？→ 內部還有其他流程使用，漸進式淘汰

### Result（成果）
- **零停機完成遷移**，使用者在過渡期間完全無感
- 成功統一使用 Firebase Auth (SSO)，免除雙重帳密管理的摩擦
- 前端獲得 100% UI 自定義自由度，報表與儀表板的互動性大幅提升
- 稽核日誌改存 Firestore，達成真正的永久追溯

## 案例三：Excel 匯入精靈 — 把混亂資料變結構化

### Situation（情境）
公司的專案通訊錄以 Excel 管理，每個專案一份 Excel，格式不統一。常見問題：
- 公司欄位有合併儲存格（一家公司跨多人）
- 電話欄位混雜市話、手機、分機（同一格）
- 地址格式五花八門（有的有郵遞區號，有的沒有）
- 同一人出現在多個專案 Excel，重複建檔

### Task（任務）
設計一個匯入精靈，讓非技術人員能上傳 Excel，系統自動處理混亂資料並匯入。

### Action（行動）

**設計：4 步驟精靈（降低認知負荷）**

1. **Step 1: 上傳 + 專案偵測**
   - 從檔名自動偵測專案代號（正則比對）
   - 呼叫 API 搜尋對應專案，自動填充

2. **Step 2: 欄位對應**
   - 關鍵字自動匹配（「姓名」→ name、「公司」→ company）
   - 未匹配的欄位提供下拉手動選擇
   - 偵測合併儲存格，自動向下填充

3. **Step 3: 預覽 + 資料清洗**
   - 電話自動分離：`"04-2213-8848 #851\n0918-774-526"` → 市話 + 分機 + 手機
   - 地址自動解析：提取郵遞區號、縣市、街道
   - 公司名稱切分：`"集鈞顧問/集登設計"` → 兩家公司
   - 重複檢測：比對現有聯絡人，標記可能的重複

4. **Step 4: 確認 + 執行**
   - 顯示統計：新增 X 人、關聯既存 Y 人、新建公司 Z 家
   - 批次匯入 + 進度顯示

**關鍵演算法**：

- **公司名稱切分**：支援 `/`、`、`、空格分隔，但合併短英文前綴（避免 `ABC Corp` 被拆成 `ABC` + `Corp`）
- **姓名正規化**：移除職稱後綴（經理、工程師等）、移除括號附註
- **重複比對**：正規化姓名 + 正規化公司名稱 → 精確匹配

### Result（成果）
- 非技術人員 **5 分鐘內** 可完成一份 Excel 匯入（過去需要 1-2 小時手動輸入）
- 電話格式統一，地址結構化
- 重複建檔率大幅降低
- 公司與聯絡人的多對多關係自動建立

## 案例四：合併引擎 — 處理歷史資料的重複問題

### Situation（情境）
系統中累積了 3,000+ 筆聯絡人，由於歷史原因（多人各自輸入），存在大量重複：
- 同一人，名字打法不同（王大明 / 王 大明 / 王大明 經理）
- 同一公司，不同寫法（台積電 / 台灣積體電路）
- 部分資料完整，部分殘缺

手動逐筆清理不現實，需要批量合併工具。

### Task（任務）
設計一套合併引擎，讓管理員可以選擇 2-5 筆重複記錄，智慧合併為一筆。

### Action（行動）

**合併流程設計**：

1. **欄位自動分類**
   ```
   衝突欄位：多筆記錄值不同 → 下拉選單讓使用者選
   自動繼承：部分有值部分空 → 自動選非空值
   相同欄位：全部一樣 → 直接保留
   ```
   → 90% 的欄位不需要使用者操作，大幅降低工作量

2. **關係繼承**
   - 合併聯絡人時：轉移所有「專案-聯絡人」關聯到主記錄
   - 合併公司時：更新所有旗下聯絡人的公司指向、更新專案關聯
   - 合併集團時：更新所有旗下公司的集團指向
   - 自動檢查重複關聯（同一專案不能關聯兩次同一人）

3. **軟刪除 + 可追溯**
   - 被合併的記錄不真正刪除，而是標記 `mergedIntoId`
   - 稽核日誌記錄完整操作（誰合併了誰、選了哪些欄位）
   - 未來可回溯或還原

4. **預覽確認**
   - 合併前顯示最終結果預覽
   - 變更的欄位用視覺標示（粗體 = 已變更、斜體 = 已清除）
   - 紅色警示：「以下記錄將被封存」

**技術難點**：
- 空值判斷：`{}` 和 `{ city: "", zip: "" }` 都應視為空
- 陣列欄位（如多支電話）：允許分別選擇第 1、第 2 支
- Firestore 批次操作：合併涉及多個 Collection 的更新，需要原子性

### Result（成果）
- 管理員 **1 分鐘內** 可完成一組合併（過去需要手動比對 + 修改 + 刪除，約 15 分鐘）
- 所有操作可追溯，不怕誤合併
- 成功清理了數百筆歷史重複資料

## 案例五：認證與稽核系統 — 企業級安全基礎

### Situation（情境）
系統初期沒有認證機制，任何知道網址的人都可以存取和修改資料。隨著系統上線使用，需要：
- 限制只有公司員工可以存取
- 區分不同權限等級（管理員 vs 一般使用者）
- 記錄所有操作（誰在什麼時候做了什麼），供事後追查

### Task（任務）
在不影響現有功能的前提下，加入完整的認證、授權、稽核三層安全機制。

### Action（行動）

1. **認證層：Firebase Authentication**
   - Google OAuth 登入（員工已有公司 Google 帳號）
   - 網域限制：只允許 `@company-domain.com` 帳號
   - Token 自動注入 HTTP 請求（Axios Interceptor）
   - Token 驗證快取（LRU Cache, 1000 筆）避免重複驗證

2. **授權層：RBAC 角色控制**
   ```
   admin: 合併、刪除、使用者管理、稽核日誌查看
   user:  CRUD 操作、匯入匯出
   viewer: 只讀
   ```
   - 後端：FastAPI 依賴注入 `Depends(require_role(["admin"]))`
   - 前端：`<RoleGuard>` 元件隱藏無權限按鈕
   - 使用者封存/恢復功能（離職員工處理）

3. **稽核層：全量操作日誌與反向還原防護**
   - 每個 CRUD 操作自動記錄
   - 記錄內容：操作者 Email、IP 位址、操作類型、變更前後差異
   - Firestore 儲存（永久保存，刻意不設過期機制，以確保合併失誤時能寫腳本反向還原）
   - 管理員可查看、篩選、匯出日誌

### Result（成果）
- **建立標準訪問控制**：在正式推廣前，成功整合 JWT 與 RBAC 機制，確保核心 API 不受未經授權的存取。
- 操作完全可追溯：曾成功追查到內部測試時的一筆誤刪資料，透過稽核日誌找到操作者和時間
- 員工離職時一鍵封存帳號，不影響歷史日誌
