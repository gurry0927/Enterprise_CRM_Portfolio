# 作品案例集 - STAR Method

> 企業聯絡人管理系統（CRM）| 獨立全端開發 | 45 天

---

## 案例一：LINE Bot AI 名片掃描 — 終結手動打字地獄

**痛點還原：**
當時我看到業務跑完外面，回公司還要花一兩小時對著鍵盤 key 名片。一張名片平均花 3-5 分鐘，且經常出現錯字、格式不一致、甚至重複建檔等問題。公司擁有 3,000+ 筆聯絡人，每週新增 10-20 張名片，這個隱形成本高得嚇人。

**我的解鈴方案與架構收編：**
當時同事已經在舊的 Google Apps Script (GAS) 環境下寫出了一個非常棒的名片辨識 PoC (概念驗證)。我的任務，是將這位同事的心血結晶從脆弱的 GAS 環境中拔出來，進行模組化重構，並完美對接到我剛建立的 Python (FastAPI) + Firestore 企業級全新架構中，讓它具備阻擋重複建檔的防呆機制。

**技術實踐與整合決策：**
- **承襲優良設計**：保留同事選用的 LINE Bot 與 AI Vision API (GPT-4o) 組合，因為這是零學習成本且辨識率最高的方案。
- **重構辨識引擎**：我接手後，優化了結構化提示詞，支援 18 個欄位的智慧提取，並內建信心分數機制，低於 50% 的欄位一律交由人工確認。
- **針對台灣情境打磨**：大幅強化了市話/手機的自動正規化、+886 國際碼轉換與郵遞區號提取邏輯。

**2. 導入「生產級」的智慧重複比對**
   - 設計「公司+姓名」組合比對策略（同名但不同公司 → 不同人）
   - 電話正規化後比對（02-1234-5678 = 0212345678 = +886-2-1234-5678）
   - 找到重複時，用差異卡片展示新舊資料對比，讓使用者決定

3. **互動式確認流程**
   ```
   拍照 → AI 辨識 → 預覽卡片（可修改）→ 重複檢查 → 確認寫入
   ```
   - Quick Reply 按鈕讓使用者能快速編輯特定欄位
   - Carousel 卡片同時展示多個重複候選人

**架構決策**：
- 為什麼用 LINE 不用 Web App？→ 業務在外面沒有電腦，LINE 最順手
- 為什麼用 AI Vision 不用 Google OCR？→ 名片排版千變萬化，傳統 OCR 準確率不足
- 為什麼做重複比對而非直接建立？→ 業務常拿到同一人多次名片

**最終業務成效：**
我大約花了 **3 天** 完成這個 PoC 的架構移植與雙向整合。實測結果，名片建檔時間從原本的 **3-5 分鐘暴降至 10 秒**。這不僅成功打通了「現場拍照 → 系統建檔」的自動化流程，也展現了我將團隊的實驗性程式碼（PoC）重構為 Production-ready 企業級應用的工程能力。

## 案例二：資料庫遷移 — 在飛行中換引擎的平滑過渡

**架構的先天瓶頸：**
系統最初是建立在原有的 Odoo ERP 平台上。但隨著前端應用擴展（LINE Bot、React Web App），營運團隊與我面臨了兩大系統瓶頸：一是 Odoo 原生的多重認證機制造成員工需頻繁登入不同系統；二是其僵化的視圖框架，根本無法滿足高階主管對客製化報表的 UI 需求。
此外，早期的稽核日誌存放在容器內的 SQLite，只要容器一重啟，日誌就會直接蒸發。

**我的破局之策：**
我向團隊提案，將核心聯絡人資料與前後端解耦，大膽遷移至雲端資料庫（Firestore），一舉解決認證摩擦並解放 UI 開發彈性。但真正的工程挑戰在於：我們必須在系統即將交由 IT 內部測試的前夕，在不弄壞任何一筆既有高價值聯絡人資料的前提下，無損地完成這項「換心手術」。

**遷移策略與技術實踐：**(漸進式，非 Big Bang)

1. **Schema 設計**
   ```
   /groups          21 筆（集團）
   /companies    1,014 筆（公司）
   /contacts     1,784 筆（聯絡人）
   /projects     2,019 筆（專案）
   /project_contacts    （多對多關聯）
   /audit_logs          （稽核日誌，新建）
   ```
   - 設計 ID 映射規則：ERP ID → Firestore Document ID
   - 保留原始 ID 以便回溯

2. **遷移腳本**
   - 批次讀取 ERP 資料 → 清洗轉換 → 寫入 Firestore
   - 處理欄位名稱轉換（ERP 使用 `x_` 前綴）
   - 處理關聯資料的一致性

3. **雙軌運行期（保證退路）**
   - 環境變數控制資料來源：`DATA_SOURCE=firestore` / `DATA_SOURCE=odoo`
   - API 層維持完全相同的介面
   - 逐步將讀寫切換至新引擎，確認無誤後才正式切斷舊系統依賴

4. **前端狀態同步與快取優化**
   - 導入「樂觀更新 (Optimistic UI)」：操作瞬間重繪畫面，隱藏 API 延遲
   - 設計自癒快取 (Self-Healing Cache)：透過全域攔截器自動清理被刪除的「幽靈狀態」
   - 經由前端快取與寫入防護機制，大幅降低資料庫的冗餘讀取次數

5. **稽核日誌遷移**
   - SQLite → Firestore（永久儲存，不設自動清理）
   - 確保所有資料合併操作皆可透過日誌腳本反向溯源與還原
   - 編寫歷史資料遷移腳本

**關鍵決策**：
- 為什麼選 Firestore？→ 全託管、自動備份、即時讀寫、免運維
- 為什麼不一次全切？→ 降低風險，出問題可以立刻切回
- 為什麼保留 ERP？→ 內部還有其他流程使用，漸進式淘汰

**落地成果：**
- **在內部封閉測試期完美完成底層引擎的替換**，避免了系統正式上線後才面臨架構僵化的超級災難。
- 成功統一使用 Firebase Auth (SSO)，免除雙重帳密管理的摩擦。
- 前端獲得 100% UI 自定義自由度，報表與儀表板的互動性巨幅提升。
- 稽核日誌正式改存 Firestore，達成真正的永久追溯。

## 案例三：Excel 匯入精靈 — 馴服混亂資料的自動化工程

**營運痛點：**
公司的專案通訊錄是以多種非標準化的 Excel 表格管理的。每個專案一份，格式極度發散：
- 公司欄位充斥著跨多人的「合併儲存格」。
- 同一格欄位裡，市話、手機、分機混雜儲存。
- 地址更是有些有郵遞區號、有些沒有。
- 不同專案的 Excel 把同一個人建了無數次檔案。

**我的解鈴方案與後端賦能：**
同事事前已經很有遠見地開發出了包含 4 個步驟的「匯入精靈 UI」概念。我接手這個 PoC 後，專注於**為這個漂亮的 UI 裝上強大的資料處理引擎**。我的任務是處理最困難的「內容清洗、與資料庫既有實體的智慧比對、以及最終安全的關聯入庫」，讓系統自動完成繁瑣的資料正規化，釋放營運人員的手動建檔時間。

**分工與核心防呆設計：**

- **前端 UI (同事設計)**：負責引導使用者上傳、欄位對應與最終確認，降低認知負荷。
- **後端清洗引擎 (我主導實作)**：
   1. **資料清洗與正規化**：將 `"04-2213-8848 #851\n0918-774-526"` 自動拆解為市話、分機與手機；並將地址自動解析出郵遞區號與縣市。
   2. **公司名稱切分與實體建立**：處理 `集鈞/集登` 這種多公司合併字串，自動在資料庫拆成兩家獨立公司。
   3. **防禦性重複比對**：在寫入資料庫前，將清洗後的姓名與公司名稱與現有聯絡人進行交叉比對，標出衝突。
   4. **多對多關係網建立**：在確認匯入時，確保人和公司的關聯（ProjectContact）能被正確建立且不重複寫入。

**關鍵演算法**：

- **公司名稱切分**：支援 `/`、`、`、空格分隔，但合併短英文前綴（避免 `ABC Corp` 被拆成 `ABC` + `Corp`）
- **姓名正規化**：移除職稱後綴（經理、工程師等）、移除括號附註
- **重複比對**：正規化姓名 + 正規化公司名稱 → 精確匹配

**解放人力的成果：**
現在，非技術人員在 **5 分鐘內** 即可無痛完成一份包含數百筆混亂聯絡人的 Excel 匯入（過去這需要 1-2 小時的手動修剪整理）。不僅電話與地址全數正規化，更透過智慧比對大幅壓低了重複建檔率。

## 案例四：合併引擎 — 處理歷史共業的數位手術刀

**歷史遺留的技術債：**
系統內積累了 3,000+ 筆聯絡人，由於早年多人各自隨意建檔，出現了大量潛藏的語意性重複：
- 同一人寫法百百種（王大明 / 王 大明 / 王大明 經理）。
- 公司名稱千奇百怪（台積電 / 台灣積體電路）。
要叫人手動一筆筆清，根本是不可能的任務。

**我的解鈴方案：**
為了根除病灶，我設計了一套半自動的合併引擎，賦能管理員能一次勾選多筆重複記錄，智慧濃縮為一筆「權威資料」。

**核心清洗設計：**

1. **欄位自動分類**
   ```
   衝突欄位：多筆記錄值不同 → 下拉選單讓使用者選
   自動繼承：部分有值部分空 → 自動選非空值
   相同欄位：全部一樣 → 直接保留
   ```
   → 90% 的欄位不需要使用者操作，大幅降低工作量

2. **關係繼承**
   - 合併聯絡人時：轉移所有「專案-聯絡人」關聯到主記錄
   - 合併公司時：更新所有旗下聯絡人的公司指向、更新專案關聯
   - 合併集團時：更新所有旗下公司的集團指向
   - 自動檢查重複關聯（同一專案不能關聯兩次同一人）

3. **軟刪除 + 可追溯（秉持 [Data Cleaning with Traceability](https://github.com/gurry0927/data-cleaning-with-traceability) 設計哲學）**
   - 被合併的記錄不真正刪除，而是標記 `mergedIntoId`
   - 稽核日誌記錄完整操作（誰合併了誰、選了哪些欄位）
   - 將「資料清洗」視為「建立映射 (Mapping)」而非「覆寫 (Overwrite)」，確保未來可完美回溯或還原

4. **預覽確認**
   - 合併前顯示最終結果預覽
   - 變更的欄位用視覺標示（粗體 = 已變更、斜體 = 已清除）
   - 紅色警示：「以下記錄將被封存」

**技術難點**：
- 空值判斷：`{}` 和 `{ city: "", zip: "" }` 都應視為空
- 陣列欄位（如多支電話）：允許分別選擇第 1、第 2 支
- Firestore 批次操作：合併涉及多個 Collection 的更新，需要原子性

**深層清洗與除錯的成果：**
現在，管理員平均只要 **1 分鐘** 就能完成一組複雜的多實體合併作業（過去得花 15 分鐘人工比對、手動複製貼上與刪除）。在這套引擎的加持下，我們已經安全地清創了數百筆歷史髒資料，且全程 100% 可追溯。

## 案例五：RBAC 認證與稽核 — 建立企業級安全底盤

**系統未設防的營運風險：**
系統原型剛跑起來時，完全處於「信任模式」，知道網址等於擁有全世界。當系統要正式推送給全公司使用時，絕不能繼續當個沒有門禁的大型試算表。
需要區分管理員與一般使用者的權限邊界，並且每一筆 CRUD 操作都必須留下「誰、在什麼時候、改了什麼」的鐵證。

**我的解鈴方案：**
在完全不干擾前端營運體驗的前提下，我以插拔式的設計，無縫崁入了包含 JWT 認證、RBAC 角色授權、以及全量稽核日誌的三層安全裝甲。

**技術實踐與決策：**

1. **認證層：Firebase Authentication**
   - Google OAuth 登入（員工已有公司 Google 帳號）
   - 網域限制：只允許 `@company-domain.com` 帳號
   - Token 自動注入 HTTP 請求（Axios Interceptor）
   - Token 驗證快取（LRU Cache, 1000 筆）避免重複驗證

2. **授權層：RBAC 角色控制**
   ```
   admin: 合併、刪除、使用者管理、稽核日誌查看
   user:  CRUD 操作、匯入匯出
   viewer: 只讀
   ```
   - 後端：FastAPI 依賴注入 `Depends(require_role(["admin"]))`
   - 前端：`<RoleGuard>` 元件隱藏無權限按鈕
   - 使用者封存/恢復功能（離職員工處理）

3. **稽核層：全量操作日誌與反向還原防護**
   - 每個 CRUD 操作自動記錄
   - 記錄內容：操作者 Email、IP 位址、操作類型、變更前後差異
   - Firestore 儲存（永久保存，刻意不設過期機制，以確保合併失誤時能寫腳本反向還原）
   - 管理員可查看、篩選、匯出日誌

**防護成果：**
在系統正式橫向推廣前，這套防護網成功鎖死了一切未經授權的 API 調用。最核心的價值體現在一次內部測試中：某位測試員誤刪了重要資料，我在五分鐘內透過稽核日誌完美鎖定了操作軌跡，證明了這套系統真正具備了企業級的溯源修復能力。
